Data flow

Two end-to-end walkthroughs: the life of a single job, and the life of a batch. Follow the actual function calls.

Part 1 — the life of a job

A job is one prompt run against one project. Here is every hop, from "queue it" to "it is saved".

caller                  Netra::queue_job(project_id, prompt)
  |                         |  builds a Job (status = Queued)
  |                         |  JobRepo::create(job)           -> SQLite row
  |                         |  JobQueue::enqueue(job)         -> in-memory queue
  |                         |  EventBus::publish(JobQueued)
  |                         v
Scheduler::run loop      acquires a concurrency permit (semaphore)
  |                      JobQueue::dequeue()  (or wait_for_job() if empty)
  |                      SpecResolver::resolve(job) -> AgentSpec
  |                      spawns a task: JobRunner::run(job, spec, cancel)
  |                         v
JobRunner::run           JobRepo::update(job, status = Running)
  |                      EventBus::publish(JobStarted)
  |                      loop up to RetryPolicy.max_attempts:
  |                         attempt(job, spec, cancel)
  |                         v
JobRunner::attempt       AgentPool::acquire(project, spec) -> AgentLease
  |                      AgentTransport::send_prompt(request) -> event stream
  |                      for each AgentEvent:
  |                         dispatch():  OutputSink::on_event()  -> log file
  |                                      EventBus::publish(JobProgress)
  |                                      fold event into (text, usage)
  |                      OutputSink::flush()
  |                         v
JobRunner::run           on success: JobRepo::update(Completed, output)
  |                                   EventBus::publish(JobCompleted)
  |                      on Transport error + retries left: sleep, retry
  |                      on cancel:    JobRepo::update(Cancelled)
  |                      on failure:   JobRepo::update(Failed)
  v                                   EventBus::publish(JobFailed)

Step by step

  1. Queue. Netra::queue_job builds a Job, persists it through JobRepo, pushes it onto the InMemoryJobQueue, and publishes a JobQueued event.
  2. Schedule. The Scheduler runs a permanent loop. It first acquires a semaphore permit — this is the concurrency cap — then takes the next job. If the queue is empty it parks on wait_for_job() instead of polling.
  3. Resolve the spec. A SpecResolver turns the job into an AgentSpec — which pi binary, working directory, model, and tool policy to use. Model precedence is job > project > app default.
  4. Run. JobRunner::run marks the job Running, then calls attempt() in a retry loop.
  5. Attempt. attempt() checks an agent out of the AgentPool, sends the prompt over the AgentTransport, and consumes the resulting AgentEvent stream.
  6. Fan out events. For every event, dispatch() does three things at once: writes it to each OutputSink (the log file), publishes a JobProgress event on the EventBus, and folds the text/usage into the running AgentOutput.
  7. Finish. On success the job is saved Completed with its output. A transport error retries with exponential backoff; cancellation marks it Cancelled; any other error marks it Failed. Each terminal state publishes a matching event.

Part 2 — the life of a batch

A batch runs a series of prompts across one or more target projects, then merges the answers. It is a classic map → reduce.

Netra::create_batch(name, prompts, projects, strategy)
   |  BatchBuilder builds a Batch
   |  Batch::materialize() -> one BatchItem per (prompt x target)
   |  BatchRepo::create(batch) + add_item() for each item
   v
Netra::run_batch(batch_id)   spawns a background task:
   v
BatchExecutor::run
   |  BatchRepo::update(status = Running);  publish BatchStarted
   |
   |  --- MAP ---------------------------------------------------
   |  for each BatchItem:
   |     resolve_prompt()  (inline body OR template + {{var}} render
   |                        + optional JSON-schema instruction)
   |     JobRepo::create(job)
   |     JobQueue::enqueue(job)   <-- the SAME queue the Scheduler drains
   |     publish JobQueued
   |  (those jobs now run through Part 1, honouring the global cap)
   |
   |  collect():  poll JobRepo::get() every poll_interval until every
   |              job is terminal;  gather the AgentOutputs
   |
   |  --- REDUCE ------------------------------------------------
   |  reduce():  if strategy == "reviewer"  -> run_reviewer()
   |             else  -> AggregatorRegistry.get(strategy).aggregate()
   v
   AggregatedResult  ->  BatchRepo::update(Completed, result)
                         publish BatchCompleted

The crucial detail: one shared queue

ⓘ The BatchExecutor does not run jobs itself. Its map step enqueues jobs onto the very same InMemoryJobQueue the Scheduler is draining. That is why batch jobs automatically honour the global concurrency cap — they are ordinary jobs. The executor then just waits (polls JobRepo::get()) for them to finish.

Map: turning prompts into jobs

resolve_prompt() builds the final prompt text for each item: it loads the prompt body (inline, or fetched from TemplateRepo), substitutes {{variable}} placeholders from the target's bindings, and — if the batch has an output_schema — appends a "respond as JSON matching this schema" instruction.

Reduce: merging the answers

The --strategy picks the reduce step:

StrategyWhat it doesWhere
concat Concatenates every output verbatim under numbered headers. ConcatAggregator
schema_merge Merges each output's structured JSON into one deduplicated array. SchemaMergeAggregator
reviewer Enqueues one more agent job that consolidates, dedupes, and ranks the map outputs. BatchExecutor::run_reviewer

concat and schema_merge live in the AggregatorRegistry; reviewer is special-cased by the executor because it needs to run another agent.

Part 3 — events reaching the UI

Everything published on the EventBus can also reach the desktop UI. The path:

EventBus (BroadcastEventBus, a tokio broadcast channel)
   |  Netra::subscribe_events()
   v
src-tauri/src/lib.rs   setup hook spawns a task:
   |  for each DomainEvent:  app.emit("netra://event", event)
   v
ui/src/lib/api.js      subscribeEvents() -> listen("netra://event")
   v
ui/src/lib/store.svelte.js   startEvents() routes each event:
   |  job_progress  -> ingestAgentEvent()  (live token streaming)
   |  job_started   -> ensureAgent() + scheduleRefresh()
   |  job_completed -> finishAgent()  + scheduleRefresh()
   |  batch_*       -> scheduleRefresh()  (200 ms debounce)
   v
