Architecture

Six Rust crates plus a desktop shell and a web UI, stacked so that dependencies only ever point one way: downward.

The one rule: dependencies point down

The dependency direction is enforced and one-way:

core  <-  { agent, store, engine }  <-  api  <-  { cli, src-tauri }
  • chatur-core depends on no internal crate.
  • agent, store, engine depend only on core (and engine also on agent).
  • api wires the three capability crates together.
  • cli and src-tauri depend only on api.

Nothing lower ever imports anything higher. This is what keeps the system understandable: you can read chatur-core with no other context, then read each layer knowing it can only use what is below it.

Every crate boundary is a trait

This is the second load-bearing idea. chatur-core defines interfaces (Rust traits); the other crates provide implementations. The engine never depends on SQLite — it depends on the JobRepo trait, which SQLite happens to implement. Consequences:

  • Swappable. Any layer can be replaced without touching its callers.
  • Mockable. Tests inject fakes — MockTransport, in-memory repos — so most of the suite runs with no pi process and no real database.

The trait files are chatur-core/src/traits/; the Design patterns page maps each one to its pattern.

What each crate is responsible for

chatur-core — domain + interfaces

Pure types and traits. No process spawning, no DB, no async I/O. Defines Project, Job, Batch, AgentEvent, AgentOutput, AggregatedResult, the typed IDs, the CoreError type, and every trait the other crates implement.

chatur-agent — the pi driver

Spawns pi --mode rpc, frames its JSON-Lines protocol, maps pi events onto the normalized AgentEvent, and bounds how many agents run at once with an AgentPool.

chatur-engine — orchestration

The moving parts: a job InMemoryJobQueue, a Scheduler that drains it under a concurrency cap, a JobRunner that executes one job (with retry), a BatchExecutor that does map→reduce, and the aggregation strategies.

chatur-store — persistence

SQLite schema and migrations, repository implementations of the core repo traits, and FileLogSink which writes every job's events to a log file.

chatur-api — the facade

The single public entry point: the Chatur struct wires store + agent pool + engine together and exposes one async method per operation (add_project, queue_job, run_batch, …). Also owns TOML config loading.

chatur-cli — the headless binary

A thin clap CLI over chatur-api. Its real job is to prove the library is fully usable without any UI.

src-tauri + ui — the desktop shell

A Tauri v2 shell: one #[tauri::command] per Chatur method, plus a bridge that forwards the engine's event stream to the SvelteKit front-end as a chatur://event.

ⓘ src-tauri is a separate Cargo workspace (the root Cargo.toml excludes it) so the main build and CI do not need system webkit libraries. See the Tauri guide.
