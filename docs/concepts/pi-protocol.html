The pi RPC protocol

How netra-agent talks to the external pi coding agent — the one foreign interface in the whole system.

What pi is

pi is a separate command-line AI coding agent (pinned to v0.74.2). NETRA never reimplements an agent — it launches pi as a child process and drives it. The protocol is captured in project memory and at https://pi.dev/docs/latest/rpc.

RPC mode & framing

NETRA runs pi --mode rpc. Communication is JSON Lines (NDJSON) over stdin/stdout: one JSON object per line.

⚠ Strict LF framing. Lines are delimited by \n only. The decoder must not use a Node-readline-style splitter, because that also breaks on U+2028 / U+2029 — characters that are legal inside JSON strings. The Rust side uses BufRead::read_until(b'\n'), which is correct. See transport.rs.

Requests (NETRA → pi)

Defined in protocol.rs as the RpcRequest enum. Only two are sent:

{"type":"prompt","id":"<corr-id>","message":"<text>"}
{"type":"abort","id":"<corr-id>"}
⚠ The prompt text field is message, not prompt. Sending prompt instead produced the historical flaky error Cannot read properties of undefined (reading 'startsWith').

The id is a correlation id: pi echoes it back on the matching response so the transport can pair them.

Responses (pi → NETRA)

A command acknowledgement looks like:

{"type":"response","command":"prompt","id":"<corr-id>","success":true}
{"type":"response","command":"prompt","success":false,"error":"<why>"}

Decoded into RpcResponse { id, success, error }. The transport holds a map of pending correlation ids; the matching response unblocks the waiter.

The event stream

While a turn runs, pi emits events with no id. map_event() translates the ones NETRA cares about into the normalized AgentEvent enum. Everything else is deliberately ignored, not errored — the protocol is version-tied, so unknown messages must not crash a run.

pi event typeConditionBecomes AgentEvent
turn_start—TurnStart
turn_end—TurnEnd
message_update assistantMessageEvent.type == "text_delta" AssistantText { text }
message_update assistantMessageEvent.type == "thinking_delta" Thinking { text }
tool_execution_start— ToolCall { name, args }
tool_execution_end— ToolResult { name, is_error }
extension_error— Error { message }
agent_end— not an event — classified as TurnComplete, which ends the turn's event stream
anything else— ignored (e.g. queue_update, message_start, auto_retry_*)

How a turn plays out

RpcTransport::send_prompt
   |  write {"type":"prompt","id":"r1","message":"..."}\n  to pi stdin
   |  await the {"type":"response","id":"r1",...}  (prompt accepted?)
   |  hand back a stream of AgentEvent
   v
read_loop (background task)  reads pi stdout line by line:
   |  parse_line() -> Incoming
   |     Response   -> wake the pending correlation-id waiter
   |     Event      -> push AgentEvent into the active turn's channel
   |     TurnComplete -> close the turn channel (stream ends)
   |     Ignored    -> drop

Other moving parts

  • extension_ui_request — pi extensions can ask for UI input (select / confirm / input / editor). A headless orchestrator must auto-respond or surface it, or the run stalls. (Notify / setStatus / setWidget are fire-and-forget.)
  • stdin EOF — pi --mode rpc exits when its stdin closes, so the transport keeps the child's stdin open for the whole session.
  • Local models — the dev setup uses provider llamacpp, model qwen3.6-35b-a3b, served by a llama.cpp OpenAI-compatible server. See the Getting started guide.
