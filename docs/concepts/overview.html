Overview — explain it like I'm five

No code in this page. Just the idea of what Mini ChatUR is, what problem it solves, and the words you need to know.

The one-paragraph version

There is a command-line AI coding agent called pi. You can talk to it, and it can read and edit code. Talking to it one question at a time, in a terminal, is fine — but it does not scale. You cannot easily run ten questions at once, keep a history, run the same review across a whole project, or watch several agents work side by side. Mini ChatUR is the control room for pi. It queues your questions, launches agent processes for you, streams their output live, saves every run, and can fan one set of prompts out across a project and merge the results.

The problem, concretely

Say you want an AI to review your repository. You might want it to:

  • find logic bugs,
  • find security vulnerabilities,
  • find missing tests,
  • suggest refactors —

… and you want all four to run, ideally at the same time, and you want one tidy report at the end instead of four messy transcripts. Doing that by hand means babysitting four terminals. Mini ChatUR does it as a single batch.

The three things it does

1. Queue single jobs
Ask one question against one project. It runs in the background under a concurrency limit.
2. Run batches
Run a series of prompts across a project, then combine every answer into one consolidated result (map → reduce).
3. Log everything
Every event from every run is saved to SQLite and to per-job log files you can inspect later.

The key words

You will see these everywhere. The full list is in the Glossary; these five are the ones you cannot skip:

WordWhat it means
pi The external AI coding-agent CLI that Mini ChatUR drives. Mini ChatUR never is the AI — it controls it.
Project A local code folder you have registered. The agent operates inside it.
Job One prompt run against one project by one agent process. The smallest unit of work.
Batch Many prompts run across one or more projects, then aggregated into a single result.
Aggregator The "reduce" step — the strategy that merges all the batch's answers (concatenate, merge JSON, or have a reviewer agent consolidate them).

How a request travels (the 30-second tour)

You click "Find Bugs"  (or run `chatur batch run ...`)
        |
        v
  chatur-api          creates a Batch, turns it into Jobs
        |
        v
  chatur-engine       a queue holds Jobs; a scheduler pulls them
        |              under a concurrency cap; a runner executes each
        v
  chatur-agent        spawns a real `pi` process, talks RPC to it
        |
        v
  pi  +  your code    the agent reads/edits files, streams events back
        |
        v
  events fan out  ->  log files + SQLite + a live event bus -> the UI
        |
        v
  aggregation         all answers merged into one AggregatedResult

Why it is split into so many crates

Mini ChatUR is deliberately built as six small Rust crates instead of one big program. Each crate has one job, and they are stacked so that the lower ones never know about the higher ones. That makes every layer testable in isolation (you can swap a real agent for a fake one) and means the desktop UI is genuinely optional — the chatur command-line tool proves the whole thing works without it.
