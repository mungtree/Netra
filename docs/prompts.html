Maintenance prompts

Ready-to-paste prompts for telling an agent how to update and maintain the Mini ChatUR codebase — and this wiki. Copy a block, fill the <ANGLE BRACKETS>, send.

These encode the project's patterns so an agent stays consistent with the architecture. Pair them with AGENTS.md (for the wiki) and the Design patterns page.

Part A — maintaining the codebase

A1. Add a new aggregation strategy

The reduce step is a Strategy. A new pure strategy is one trait impl plus one registry line.

Add a new aggregation strategy named "<STRATEGY_ID>" to chatur-engine.

1. In crates/chatur-engine/src/aggregate.rs, add a unit struct and
   `impl Aggregator` for it (trait: crates/chatur-core/src/traits/aggregator.rs).
   Implement id() -> "<STRATEGY_ID>", description(), and async aggregate(),
   following ConcatAggregator / SchemaMergeAggregator in the same file.
2. Register it in AggregatorRegistry::with_defaults() alongside the others.
3. Re-export the new struct from crates/chatur-engine/src/lib.rs.
4. Add a #[tokio::test] in the aggregate.rs tests module.
5. Update the docs: crates/chatur-engine reference page aggregate.html, and the
   strategy table in docs/concepts/data-flow.html.

Do NOT special-case it in BatchExecutor unless it must run a live agent (only
the "reviewer" strategy does that). Run `cargo test -p chatur-engine`.

A2. Add a new operation to the public API

A new capability must travel the full stack: Chatur method → CLI &/or Tauri command → api.js wrapper.

Add a new operation "<OPERATION>" to Mini ChatUR, end to end.

1. crates/chatur-api/src/chatur.rs — add an async method on `Chatur` that does
   the work through the existing repos / engine. Match the style of queue_job /
   create_batch. Return chatur_core::Result<T>.
2. crates/chatur-cli/src/main.rs — if it should be CLI-reachable, add a clap
   subcommand and a dispatch arm.
3. src-tauri/src/commands.rs — add a #[tauri::command] that parses string ids,
   calls the Chatur method, and maps errors to String. Register it in the
   `tauri::generate_handler!` list in src-tauri/src/lib.rs.
4. ui/src/lib/api.js — add one thin async wrapper that `invoke()`s the command.
   Components must call api.js, never `invoke` directly.
5. If the UI needs it, add a store action in ui/src/lib/store.svelte.js.
6. Docs: update reference pages chatur-api/chatur.html, src-tauri/commands.html,
   ui/api.html — and the relevant Ripple-effects callouts.

Keep Tauri commands 1:1 with Chatur methods. Run `cargo test --workspace`.

A3. Add a new output sink

Add a new OutputSink "<SINK_NAME>" (e.g. a webhook / metrics sink).

1. Implement the OutputSink trait (crates/chatur-core/src/traits/sink.rs):
   id(), async on_event(), async flush(). Put it in a new file under
   crates/chatur-store/src/ (model it on sink.rs / FileLogSink).
2. Re-export it from crates/chatur-store/src/lib.rs.
3. Wire it into the JobRunner's `sinks` vec where the runner is constructed
   (crates/chatur-api/src/chatur.rs, in start_with_factory).
4. Remember on_event is awaited inline in the job event loop — keep it fast or
   buffer internally.
5. Docs: add a reference page under docs/reference/chatur-store/, a nav.js entry,
   and update the OutputSink implementors list in traits-sink.html.

A4. Add a database migration

Add a SQLite migration to chatur-store.

1. Create crates/chatur-store/migrations/000N_<name>.sql with the next number.
   Migrations run automatically on Database::connect via the embedded MIGRATOR.
2. Remember the storage model: entities are a JSON `data` blob plus promoted
   columns. You only need a migration for new indexes, columns used in queries,
   or foreign keys — NOT for new struct fields.
3. If you add a promoted column, update the matching repo in
   crates/chatur-store/src/repo/ to bind it on create/update.
4. Add/extend a test in crates/chatur-store/tests/store.rs.
5. Docs: update docs/reference/chatur-store/migration-0001.html (or add a page
   for the new migration) and the schema notes.

A5. Add a UI component

Add a new Svelte component "<Name>.svelte" to the desktop UI.

1. Create ui/src/lib/components/<Name>.svelte (Svelte 5 runes — see existing
   components like QueuePanel.svelte for the style).
2. Take data in via `$props()`; never call Tauri `invoke` directly — read from
   the store (ui/src/lib/store.svelte.js) or accept callbacks as props.
3. Render it from ui/src/routes/+page.svelte, passing the derived state it needs.
4. Reuse design tokens from ui/src/app.css; no new colour values.
5. Docs: add a reference page docs/reference/ui/<name>.html, a nav.js entry,
   and a row in docs/reference/ui/index.html.

A6. Run the checks

Run the Mini ChatUR test + lint suite and report failures.

  cargo fmt --check
  cargo clippy --workspace -- -D warnings
  cargo test --workspace

The workspace suite uses MockTransport + in-memory DB — no `pi` process needed.
For a real end-to-end agent run (needs the model server up):
  cargo test -p chatur-agent --test real_pi -- --ignored --nocapture
Note: src-tauri is a separate workspace — `cd src-tauri && cargo fmt --check`.

Part B — maintaining this wiki

Full rules: AGENTS.md.

B1. Document a newly added source file

Document the source file <PATH/TO/file.rs> in the docs/ code-wiki.

Read docs/AGENTS.md first. Then:
1. Read the source file fully — do not rely on memory.
2. Copy docs/_TEMPLATE.html to docs/reference/<crate>/<file>.html
   (flatten subdirs with `-`: model/job.rs -> model-job.html).
3. Set data-root="../.." and data-page accordingly; replace every REL.
4. Fill the 7-section reference template from AGENTS.md §4. Document every
   public item with signature + file:line. The Ripple-effects callout is
   mandatory.
5. Add a nav.js entry in the right "Reference · <crate>" section.
6. Add a row to docs/reference/<crate>/index.html.
7. Open the page in a browser and run the AGENTS.md §7 checklist.

B2. Update a reference page after a code change

The source file <PATH/TO/file.rs> changed. Update its wiki page
docs/reference/<crate>/<file>.html to match.

1. Diff the file against what the page documents.
2. Update changed signatures, descriptions, and file:line numbers.
3. Add cards for new public items; remove cards for deleted ones.
4. Re-check the Ripple-effects callout — if a cross-module call was added or
   removed, update it here AND on the other file's page AND in
   docs/concepts/cross-module-map.html.
5. Run the AGENTS.md §7 checklist.

B3. Audit the wiki for stale line numbers

Audit every docs/reference/**/*.html page against current source.

For each reference page, open the source file it documents and verify every
`file:line` in an .api-loc span still points at the right item. Fix the stale
ones. Report any function documented but no longer present, or present but not
documented.

B4. Add a reference section for a new crate

A new crate <crate-name> was added. Give it a wiki reference section.

1. Create docs/reference/<crate-name>/index.html — a crate overview (purpose,
   file table, dependencies), modelled on docs/reference/chatur-core/index.html.
2. Document each source file as its own page (prompt B1).
3. Add a new "Reference · <crate-name>" section to the SITE array in
   docs/assets/nav.js.
4. Add the crate to the crate table on docs/index.html and to the layered
   diagram on docs/concepts/architecture.html.
