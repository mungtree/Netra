netra-store / migrations/0001_init.sql

The initial database schema: five tables, four indexes, and the JSON-blob storage model.

Path: crates/netra-store/migrations/0001_init.sql

In plain terms

One-shot SQL that creates the entire schema. Every entity — project, batch, job, template, and batch-item — is stored as one JSON data blob (the source of truth) plus a few promoted columns used purely for indexing and foreign-key relations. Consequence: adding a field to a domain struct needs no migration; only changes that affect querying or relations do.

Indexes

0001_init.sql:28 CREATE INDEX idx_jobs_status ON jobs (status)

Speeds up queries that filter jobs by status (e.g. "list all queued jobs").

0001_init.sql:29 CREATE INDEX idx_jobs_project ON jobs (project_id)

Speeds up "list jobs for project X" queries. Also supports the FK join.

0001_init.sql:36 CREATE INDEX idx_batch_items_batch ON batch_items (batch_id)

Speeds up "list items of batch X" queries.

Foreign-key cascade rules

0001_init.sql:23 jobs.project_id → projects: ON DELETE CASCADE

Deleting a project deletes all its jobs automatically. No orphaned job rows.

0001_init.sql:24 jobs.batch_id → batches: ON DELETE SET NULL

Deleting a batch leaves its jobs in place but clears their batch_id, so they remain queryable as standalone jobs.

0001_init.sql:33 batch_items.batch_id → batches: ON DELETE CASCADE

Deleting a batch deletes all its batch-items. The batch's own row is removed via the FK on jobs.batch_id (SET NULL) so the batch row survives for as long as any job references it; the items do not.

Design rationale

The JSON-blob storage model means the data column on every table is the single source of truth. Promoted columns (id, name, status, created_at, project_id, batch_id) exist only to support indexing, sorting, and foreign-key constraints. This design lets domain structs evolve freely — a new field on Job requires no migration, only a code change.

⚡ Ripple effects. This migration runs once at Database::connect (netra-store/db.rs). The five tables are consumed by the four Sqlite*Repo types (repo-project.rs, repo-job.rs, repo-batch.rs, repo-template.rs). Changing a column name or adding a NOT NULL constraint here will crash startup for existing databases. Future migrations must be additive only.
