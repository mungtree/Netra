chatur-engine / scheduler.rs

The Scheduler — a permanent loop that drains the queue into the runner, never exceeding a concurrency cap.

Path: crates/chatur-engine/src/scheduler.rs

In plain terms

The scheduler is the engine's heartbeat. It loops forever: grab a concurrency permit, take a job off the queue, spawn a task to run it, repeat. The permit is taken before the job, so no more than concurrency jobs are ever in flight.

scheduler.rs:61 pub async fn run(self: Arc<Self>, shutdown: CancellationToken)

The dispatch loop. Each iteration: select!s on either shutdown firing (break) or acquiring a semaphore permit; then dequeue()s a job — or, if the queue is empty, parks on wait_for_job(); then spawns a task that holds the permit and runs execute(job). When shutdown fires the loop stops but in-flight jobs are left to finish.

scheduler.rs:104 async fn execute(&self, job: Job)

Resolves the job's spec via SpecResolver, registers a fresh CancellationToken in running, calls runner.run(job, spec, cancel), then removes the token. A spec-resolution failure is logged and the job is skipped.

How many jobs are currently executing.

⚡ Ripple effects.
  • run() is spawned once as a long-lived background task by Chatur::start_with_factory; its shutdown token is fired by Chatur::shutdown.
  • cancel_running backs Chatur::cancel_job for jobs that are already running (queued jobs are cancelled via the queue instead).
  • This concurrency cap and the AgentPool's global_max are two independent throttles — both come from ConcurrencyConfig.
  • Because BatchExecutor enqueues onto the same queue, the scheduler runs batch jobs and standalone jobs through one pipeline.
