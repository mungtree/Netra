The JobRunner — executes one job end to end: agent turn, event fan-out, retry, cancellation, persistence.

Path: crates/chatur-engine/src/runner.rs

In plain terms

This is where a single job actually runs. The runner borrows an agent from the pool, sends the prompt, and consumes the resulting event stream. Every event is fanned out three ways — to log sinks, to the event bus, and folded into the accumulating answer. If the attempt fails for a transient reason it retries; whatever the outcome, the final status is saved.

runner.rs:59 pub async fn run(&self, job: Job, spec: AgentSpec, cancel: CancellationToken) -> Result<AgentOutput>

Runs a job to its terminal state. Sets status Running and publishes JobStarted; loops attempt() up to retry.max_attempts times, sleeping retry.delay_for() between retryable failures; then writes the terminal status — Completed (+ output + JobCompleted), Cancelled (+ JobFailed "cancelled"), or Failed (+ JobFailed with the error). The final status is persisted via JobRepo::update.

runner.rs:119 async fn attempt(&self, job: &Job, spec: &AgentSpec, cancel: &CancellationToken) -> Result<AgentOutput>

One execution attempt. Acquires an agent lease from the pool, sends the prompt, and loops the event stream inside a select! that also watches cancel. Each event goes through dispatch(). When the stream ends it flushes every sink; if cancelled it aborts the transport and returns CoreError::Cancelled; an Error event becomes CoreError::Agent. Finally it opportunistically parses the collected text as JSON into structured.

runner.rs:179 async fn dispatch(&self, job_id, event, text, usage, failure)

Fans one event out: calls on_event on every sink, publishes DomainEvent::JobProgress on the bus, then folds the event into the accumulating state — AssistantText appends to text, Usage adds to usage, Error sets failure.

⚡ Ripple effects.
  • Called by Scheduler::execute — one run() per dispatched job.
  • dispatch() is the origin of every JobProgress event — the start of the chain that ends at live token streaming in the UI OutputPane.
  • Sink writes (FileLogSink::on_event) are awaited inline, so a slow sink slows the job.
  • Retry is governed entirely by RetryPolicy: only CoreError::Transport failures retry.
  • The opportunistic JSON parse here is what populates AgentOutput.structured, which SchemaMergeAggregator depends on.
