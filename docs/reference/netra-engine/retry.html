netra-engine / retry.rs

The RetryPolicy — how the job runner backs off and retries a transient failure.

Path: crates/netra-engine/src/retry.rs

In plain terms

Sometimes a job fails for a silly, temporary reason — the pi process crashed. Retrying immediately would just hammer it, so the runner waits a bit, then a bit more, then a bit more (exponential backoff). This struct holds the knobs and the rule for which failures are even worth retrying.

Public types

retry.rs:9 pub struct RetryPolicy { max_attempts: u32, base_delay: Duration, max_delay: Duration }

max_attempts counts the first try too — 1 disables retrying. The default (Default impl, retry.rs:18) is 3 attempts, 500 ms base, 30 s cap.

Methods

retry.rs:31 pub fn no_retry() -> Self

A policy with max_attempts = 1 — one try, no retries.

retry.rs:42 pub fn delay_for(&self, attempt: u32) -> Duration

The backoff before retry attempt (1-based). Grows as base_delay × 2^(attempt-1), capped at max_delay. So with a 100 ms base: 100, 200, 400, 800 ms… All arithmetic is saturating, so it never overflows.

retry.rs:55 pub fn is_retryable(error: &CoreError) -> bool

Returns true only for CoreError::Transport — a crashed or unreachable pi process. Agent rejections, bad input, and cancellation are not retried. (Associated function — no self.)

⚡ Ripple effects.
  • JobRunner::run is the only caller: its retry loop calls is_retryable to decide, and delay_for to sleep between attempts.
  • The retryable set is defined by the CoreError::Transport variant. If a failure that should be retried is mapped to a different variant (or vice versa), retry behaviour silently changes — this is the tightest coupling between this file and netra-core/error.rs.
