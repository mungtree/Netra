The reduce-step strategies and the registry that holds them.

Path: crates/netra-engine/src/aggregate.rs

In plain terms

A batch ends with many answers that must become one. This file holds the two pure ways of doing that (concatenate, merge JSON) plus the registry that maps a strategy name to its implementation. The third strategy, reviewer, needs a live agent and so lives in batch.rs instead.

The built-in strategies

aggregate.rs:76 pub struct ConcatAggregator

id "concat". Its aggregate() joins every output verbatim under a === Output N === header and sums token usage. structured is left None.

aggregate.rs:114 pub struct SchemaMergeAggregator

id "schema_merge". Its aggregate() collects each output's structured JSON into one array (arrays are flattened, scalars/objects pushed whole) and drops duplicates — by the field named in config.dedup_key if present, otherwise by deep equality. The merged array becomes both structured and a pretty-printed summary.

⚡ Ripple effects.
  • BatchExecutor::reduce calls registry.get(strategy) — an unknown strategy id becomes CoreError::Invalid and fails the batch.
  • SchemaMergeAggregator only works if upstream jobs actually produced structured JSON — which depends on the batch having an output_schema and the agent obeying it. Outputs with no structured value are silently skipped.
  • To add a strategy: implement Aggregator and register it — see the maintenance prompts.
