chatur-core / model/agent.rs

The normalized vocabulary for talking to an agent — events, token counts, and the final output.

Path: crates/chatur-core/src/model/agent.rs

In plain terms

pi emits messy, version-specific JSON. Mini ChatUR translates it once, at the edge, into the three clean types here. After that, no other crate ever sees a raw pi shape — they all speak AgentEvent and AgentOutput. This is what makes the agent swappable.

One normalized event emitted while an agent turn runs. Serialized with a "kind" tag in snake_case. Variants:

ToolResult { name, is_error }A tool finished.
Usage(TokenUsage)Token usage / cost reported by the provider.
TurnEndThe turn finished.

agent.rs:84 pub struct AgentOutput { text: String, structured: Option<Value>, usage: TokenUsage }

The final result of one completed job. structured is set only when the job requested a defined output schema and the text parsed as JSON.

⚡ Ripple effects.
  • AgentEvent is produced by protocol.rs::map_event, carried by AgentTransport, folded into AgentOutput by PiSession::run and JobRunner::attempt, written to disk by FileLogSink, and re-broadcast inside DomainEvent::JobProgress. Add a variant and every one of those must handle it.
  • TokenUsage's Add is what the aggregators use to sum cost across a batch.
  • AgentOutput is stored on Job::output and gathered by the BatchExecutor.
