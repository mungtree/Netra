The batch data model: a series of prompts fanned out across targets, plus the fluent builder that assembles one.

Path: crates/netra-core/src/model/batch.rs

In plain terms

A batch is a map/reduce job. The map inputs are its prompts × targets — every combination becomes one BatchItem (and later one job). The reduce step is its aggregation. This file is all data plus one Builder; the actual running lives in netra-engine.

⚡ Ripple effects.
  • materialize() is called by Netra::create_batch; the resulting items are persisted via BatchRepo::add_item.
  • AggregationSpec.strategy is looked up at reduce time by AggregatorRegistry::get — or special-cased to run_reviewer when it equals "reviewer".
  • output_schema, if set, makes BatchExecutor::resolve_prompt append a "respond as JSON" instruction to every prompt.
  • {{variable}} bindings in BatchTarget are substituted by BatchExecutor::render.
