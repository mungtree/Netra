netra-core / traits/repo.rs

Four storage interfaces — the Repository pattern. One trait per entity type.

Path: crates/netra-core/src/traits/repo.rs

In plain terms

The engine needs to save and load projects, jobs, batches and templates. It does not want to know about SQLite. So each entity gets a repository trait — a plain set of CRUD methods. netra-store provides the SQLite implementations; tests can provide in-memory ones.

Traits

repo.rs:14 #[async_trait] pub trait ProjectRepo: Send + Sync

Storage for Projects.

  • create(&self, project: &Project) -> Result<()> — insert.
  • get(&self, id: ProjectId) -> Result<Project> — fetch by id.
  • list(&self) -> Result<Vec<Project>> — all projects.
  • delete(&self, id: ProjectId) -> Result<()> — remove.
repo.rs:27 #[async_trait] pub trait JobRepo: Send + Sync

Storage for Jobs.

  • create / get(id) / update(&job) — insert, fetch, persist changes.
  • list_by_status(status) -> Result<Vec<Job>> — jobs in one lifecycle state.
  • list_by_project(project_id) -> Result<Vec<Job>> — every job for a project.
repo.rs:42 #[async_trait] pub trait BatchRepo: Send + Sync

Storage for Batches and their items.

  • create / get(id) / update(&batch) / list() — batch CRUD.
  • add_item(&item) / update_item(&item) — insert / persist a BatchItem.
  • items(batch_id) -> Result<Vec<BatchItem>> — a batch's items.
repo.rs:61 #[async_trait] pub trait TemplateRepo: Send + Sync

Storage for PromptTemplates: create, get(id), list().

Implementations

All in netra-store: SqliteProjectRepo, SqliteJobRepo, SqliteBatchRepo, SqliteTemplateRepo.

⚡ Ripple effects.
  • JobRepo::update is the single chokepoint for job-status persistence — called by JobRunner on every transition and polled by BatchExecutor::collect.
  • The Netra facade holds these repos (via the Database) and most of its methods are thin wrappers over them.
  • Add a method to a repo trait and every SQLite impl must implement it.
