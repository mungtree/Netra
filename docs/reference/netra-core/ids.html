netra-core / ids.rs

Strongly-typed identifier newtypes — one per entity, so the compiler rejects mixing them up.

Path: crates/netra-core/src/ids.rs

In plain terms

Every entity has a UUID. But a bare Uuid for a job looks exactly like a Uuid for a project — nothing stops you passing the wrong one. So each id gets its own tiny wrapper type. Pass a JobId where a ProjectId is expected and the code will not compile. All six wrappers are generated by one macro so they behave identically.

The macro

ids.rs:10 macro_rules! typed_id!

Declares a pub struct Name(pub Uuid) with a fixed set of derives (Debug, Clone, Copy, PartialEq, Eq, Hash, Serialize, Deserialize) and the methods/impls below. Every id type is one typed_id! invocation.

The id types

Six newtypes, all with the identical surface:

TypeIdentifies
ProjectIdA registered code project.
JobIdA single agent job.
BatchIdA batch.
BatchItemIdA batch item.
TemplateIdA prompt template.
RunIdOne execution of a job or batch.

id.to_string() yields the plain UUID string — this is what gets stored in SQLite and printed by the CLI.

ids.rs:42 impl FromStr — from_str(s) parses a UUID string

Inverse of Display; errors with uuid::Error. Used by the CLI and Tauri commands to turn string arguments back into typed ids.

⚡ Ripple effects.
  • Display/FromStr are the (de)serialization seam at every text boundary: the CLI and the Tauri commands call .parse() on user-supplied id strings.
  • The Serialize/Deserialize derives let ids ride inside the JSON blobs stored by netra-store.
