RpcTransport — owns one live pi child process and drives it.

Path: crates/chatur-agent/src/transport.rs

In plain terms

This is the real implementation of AgentTransport. It spawns pi, writes JSON-line requests to its stdin, and runs a background task that reads pi's stdout forever — routing acknowledgements back to whoever is waiting, and events into the active turn's stream.

⚠ One transport = one pi conversation. Prompts are serial: calling send_prompt while a turn is still running returns an error.

The background reader

transport.rs:186 async fn read_loop(stdout: ChildStdout, shared: Arc<Shared>)

Reads pi's stdout line by line with read_until(b'\n', ...) — strict LF framing, never a Node-readline-style splitter (which would break on U+2028/U+2029 inside JSON strings). For each line it calls parse_line and routes the result: a Response wakes the matching pending waiter; an Event is pushed to the turn sink; TurnComplete drops the turn sink (ending the caller's stream). On EOF it fails every outstanding request and closes any open turn.

⚡ Ripple effects.
  • Created by PiTransportFactory::create.
  • Its event stream is consumed by PiSession::run and by JobRunner::attempt.
  • Decoding is delegated entirely to protocol.rs.
  • The serial-turn rule means callers must finish (or drop) one event stream before sending the next prompt on the same transport.
