chatur-agent crate

The pi driver: spawns and supervises pi --mode rpc processes, frames their protocol, and bounds how many run at once.

What this crate is for

This is the one crate that knows the outside world exists. It launches the external pi binary as a child process, speaks its JSON-Lines RPC protocol, and translates pi's raw events into the normalized AgentEvent. Everything above it is provider-agnostic because of the work done here.

Depends on chatur-core only.

The layers, bottom to top

AgentSpec        how to launch one pi process
   |
RpcTransport     one pi process; send a prompt, stream events
   |  (protocol.rs frames + decodes the wire JSON)
PiSession        runs a prompt to completion -> one AgentOutput
   |
AgentPool        bounds concurrent processes (global + per-project)

Tests

  • tests/fake_pi.rs — spawns a fake pi (a Bash script) to exercise real process spawning, strict-LF framing, and id correlation in RpcTransport.
  • tests/real_pi.rs — an end-to-end test against the real pi binary and a live model server. #[ignore]d by default; run manually with cargo test -p chatur-agent --test real_pi -- --ignored --nocapture.

Who depends on it

chatur-engine (uses AgentPool + AgentSpec) and, transitively, chatur-api.
